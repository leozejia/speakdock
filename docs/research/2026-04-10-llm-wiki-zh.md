# LLM Wiki

一种使用 LLM 构建个人知识库的方法模式。

这是一份“思路文件”，设计出来就是为了被你直接复制粘贴给自己的 LLM Agent 使用，例如 OpenAI Codex、Claude Code、OpenCode / Pi 等。它的目标是传达高层次的方法论，而具体实现会由你的 Agent 与你协作共同展开。

## 核心思想

大多数人今天使用 LLM 处理文档的方式，看起来都像 RAG：你上传一批文件，LLM 在提问时检索相关片段，然后生成答案。这种方式能工作，但问题在于，LLM 每次回答问题都像是在从零重新发现知识。它没有“积累”。如果你问了一个微妙的问题，需要综合五份文档，LLM 每次都得重新去找、重新去拼接相关片段。没有任何东西被真正建起来。NotebookLM、ChatGPT 的文件上传，以及大多数 RAG 系统，基本都属于这一类。

这里的想法不一样。不是在查询时临时从原始文档里检索，而是让 LLM **渐进式地构建并维护一个持久存在的 wiki**。这个 wiki 是一组有结构、可互相链接的 Markdown 文件，位于你与原始资料之间。当你加入一个新的来源时，LLM 不是只把它索引起来以备后查，而是会真正去读它、提取关键信息，并把这些信息整合进已有 wiki：更新实体页，修订主题摘要，标记新数据与旧结论之间的矛盾，强化或挑战正在演化中的综合判断。知识被编译一次，然后持续保持最新，而不是每次提问都重新推导一遍。

这就是关键差异：**wiki 是一个持久存在、会不断复利的产物。** 交叉引用已经在那里了，矛盾已经被提前标记了，综合结论已经反映了你读过的所有内容。每增加一个新来源、每提出一个新问题，这个 wiki 都会继续变得更丰富。

你几乎永远不需要自己去写 wiki，或者至少很少需要。wiki 的写作和维护都由 LLM 完成。你负责的是选择来源、做探索、提出正确的问题。那些真正让知识库长期有用但又极其琐碎的工作，例如摘要、交叉引用、归档、记账，全部交给 LLM。实际操作里，我会把 LLM Agent 放在一边，把 Obsidian 开在另一边。LLM 会根据我们的对话去编辑内容，而我实时浏览结果，跟着链接走、看图谱视图、读更新后的页面。Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。

这个方法可以应用到很多不同场景。举几个例子：

- **个人场景**：跟踪自己的目标、健康、心理、自我提升。把日记、文章、播客笔记不断归档，逐步构建出一个结构化的自我画像。
- **研究场景**：围绕某个主题持续几周或几个月深挖，阅读论文、文章、报告，渐进式地构建一个带有持续演化论点的综合 wiki。
- **读书场景**：一章一章归档，建立人物页、主题页、情节线索页，并梳理它们之间的关系。读到最后，你会得到一个非常丰富的伴随式 wiki。可以把它想象成像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki，它由成千上万个互相链接的页面组成，覆盖人物、地点、事件、语言，多年来由志愿者社区持续维护。你也可以在个人阅读过程中做出类似的东西，只不过交叉引用和维护工作由 LLM 完成。
- **商业 / 团队场景**：一个由 LLM 维护的内部 wiki，数据源来自 Slack 讨论串、会议转录、项目文档、客户通话，也可以让人类参与审核更新。wiki 可以始终保持最新，因为那些团队里没人愿意做的维护工作交给了 LLM。
- **竞品分析、尽调、旅行规划、课程笔记、兴趣专题深挖**：凡是需要长期积累知识、并且希望这些知识被组织起来而不是散落在各处的场景，都适合。

## 架构

整个体系有三层：

**原始来源（Raw sources）**：你精心挑选的一组原始资料文件，例如文章、论文、图片、数据文件。这一层是不可变的，LLM 只会读取，不会修改。它是你的事实真相源。

**Wiki**：一个由 LLM 生成的 Markdown 文件目录。里面可能有摘要页、实体页、概念页、比较页、总览页、综合分析页。这个层级完全由 LLM 负责。新来源到来时，它会创建页面、更新页面、维护交叉引用，并尽量保持整体一致。你读 wiki，LLM 写 wiki。

**Schema**：一份说明文档，例如 Claude Code 用的 `CLAUDE.md`，或者 Codex 用的 `AGENTS.md`。它告诉 LLM 这个 wiki 应该如何组织、采用什么约定，以及在导入来源、回答问题、维护 wiki 时要遵循哪些工作流。它是关键配置文件，让 LLM 变成一个有纪律的 wiki 维护者，而不是一个泛化聊天机器人。随着你逐渐摸索出适合自己领域的方法，这份 schema 会和 LLM 一起持续演化。

## 操作

**导入（Ingest）**。你把一个新的来源放进原始资料集合里，然后告诉 LLM 去处理它。一个典型流程是：LLM 读取来源，和你讨论关键收获，在 wiki 中写一页摘要，更新索引，更新 wiki 里相关的实体页和概念页，并在日志里追加一条记录。一个来源可能会触达 10 到 15 个 wiki 页面。就我个人来说，我更喜欢一次只导入一个来源，并且自己保持参与：我会读摘要、检查更新、告诉 LLM 哪些内容更值得强调。但你也可以一次批量导入很多来源，只是监督更少。最终采用什么工作流，取决于你的风格，而这套工作流也应该写回 schema，供未来会话复用。

**查询（Query）**。你向 wiki 提问题。LLM 会先搜索相关页面，读这些页面，再带引用地综合出答案。答案形式可以根据问题不同而不同，例如一页 Markdown、一张比较表、一份 Marp 幻灯片、一个 matplotlib 图表，或者一块画布。这里有个非常重要的洞见：**好的答案可以反过来作为新页面写回 wiki。** 比如你让它做的比较、分析，或者你发现的一个新联系，这些内容本身就是有价值的，不该消失在聊天记录里。这样一来，你的探索也像导入的新来源一样，持续在知识库里复利。

**巡检（Lint）**。定期让 LLM 给 wiki 做健康检查。可以检查的内容包括：页面之间是否存在矛盾、哪些旧结论已经被新来源推翻、哪些孤儿页面没有入链、哪些重要概念被提到了却还没有独立页面、哪些交叉引用缺失、哪些数据空缺可以通过网络搜索补上。LLM 很擅长提出值得继续调查的新问题，以及值得去寻找的新来源。这能让 wiki 在成长过程中持续保持健康。

## 索引与日志

随着 wiki 变大，有两个特殊文件能帮助 LLM 和你更好地导航整个系统。它们服务于不同目的：

**index.md** 是面向内容的。它是整个 wiki 的目录。每个页面都会列出来，带上链接、一句话摘要，必要时再加上一些元数据，比如日期或者来源数。它通常按类别组织，例如实体、概念、来源等。LLM 在每次导入时都会更新它。在回答问题时，LLM 会先读 index，找到相关页面，再继续深入。这种方式在中等规模下效果 surprisingly 好，例如大约 100 个来源、几百个页面时，往往已经够用了，而且还能避免引入 embedding 式 RAG 的复杂基础设施。

**log.md** 是按时间顺序组织的。它是一个只追加的记录，记下发生过什么、发生在什么时候，例如导入、查询、lint 巡检。一个很实用的小技巧是，如果每条记录都以统一前缀开头，例如 `## [2026-04-02] ingest | Article Title`，那么这份日志就可以被简单 Unix 工具解析，比如 `grep "^## \[" log.md | tail -5` 就能直接给出最近 5 条记录。日志能帮你看到 wiki 是如何一步步演化出来的，也能帮助 LLM 理解最近做过什么。

## 可选：CLI 工具

在某个阶段，你可能会想做一些小工具，帮助 LLM 更高效地操作这个 wiki。最明显的就是给 wiki 页面做一个搜索引擎。在小规模下，index 文件已经够用；但随着 wiki 变大，你会想要更正式的搜索能力。[qmd](https://github.com/tobi/qmd) 是一个不错的选项：它是面向 Markdown 文件的本地搜索引擎，支持 BM25 / 向量混合搜索以及 LLM 重排序，而且全都可以在本地设备上运行。它既提供 CLI，也提供 MCP Server，因此 LLM 既可以通过 shell 调用它，也可以把它当作原生工具来用。你当然也可以自己做一个更简单的版本，LLM 很适合帮你快速 vibe-code 出一个朴素搜索脚本。

## 技巧与经验

- **Obsidian Web Clipper** 是一个浏览器扩展，可以把网页文章直接转成 Markdown。对于快速把来源导入到你的原始资料集合里非常有用。
- **把图片下载到本地。** 在 Obsidian 设置中的 Files and links 里，把“Attachment folder path”设成一个固定目录，例如 `raw/assets/`。然后在 Hotkeys 里搜索 “Download”，找到 “Download attachments for current file”，给它绑一个快捷键，例如 Ctrl+Shift+D。这样在剪藏一篇文章后，只要按一下快捷键，所有图片就都会被下载到本地磁盘。这不是必需的，但很有帮助，因为它能让 LLM 直接查看并引用这些图片，而不是依赖可能失效的 URL。需要注意的是，LLM 目前通常不能一次性原生读取带内嵌图片的 Markdown。比较可行的做法是让 LLM 先读文本，再单独查看其中部分或全部被引用的图片，以获取额外上下文。这个流程有点笨拙，但通常够用。
- **Obsidian 的图谱视图（graph view）** 是观察你的 wiki 结构最好的方式。它能让你看到什么和什么相连，哪些页面是枢纽，哪些页面是孤儿。
- **Marp** 是一种基于 Markdown 的幻灯片格式，Obsidian 有对应插件。它很适合直接从 wiki 内容生成演示文稿。
- **Dataview** 是 Obsidian 的一个插件，可以对页面 frontmatter 做查询。如果你的 LLM 会给 wiki 页面加 YAML frontmatter，例如标签、日期、来源数，那么 Dataview 就能动态生成表格和列表。
- 整个 wiki 本质上只是一个由 Markdown 文件组成的 Git 仓库。你天然就拥有版本历史、分支和协作能力。

## 为什么这套方法有效

维护知识库最枯燥的部分，不是阅读，也不是思考，而是记账式的维护工作。更新交叉引用、保持摘要最新、标记新数据与旧结论之间的冲突、维护几十个页面之间的一致性。人类之所以会放弃 wiki，往往是因为维护成本增长得比价值还快。LLM 不会觉得无聊，不会忘记更新交叉引用，而且一次就能改 15 个文件。于是 wiki 得以持续被维护，因为维护成本几乎接近零。

人类的工作是筛选来源、引导分析、提出好问题，并思考这些内容意味着什么。LLM 的工作则是剩下的一切。

这个想法在精神上和 Vannevar Bush 在 1945 年提出的 Memex 有关。Memex 设想的是一种个人化、经过精心整理的知识存储系统，文档之间通过联想式路径相互连接。Bush 的愿景其实比今天的 Web 更接近这里描述的方法：它是私人的、主动整理过的，而且文档之间的连接本身和文档内容一样有价值。他当年无法解决的问题，是“谁来维护这些连接和结构”。而这里，LLM 承担了这个维护者角色。

## 说明

这份文档刻意保持抽象。它描述的是一种想法，而不是某个具体实现。具体目录结构、schema 约定、页面格式、工具选型，都会取决于你的领域、偏好，以及你使用的 LLM。上面提到的所有内容都是可选且模块化的，你可以挑有用的部分，忽略没用的部分。例如：如果你的来源只有文本，那就完全不需要图片处理。你的 wiki 如果很小，可能 index 文件就已经足够了，不需要搜索引擎。你也许不关心生成幻灯片，只想要 Markdown 页面。你也许还想要一套完全不同的输出格式。正确的使用方式，是把这份文档交给你的 LLM Agent，然后和它一起把它实例化成一个适合你需求的版本。这份文档唯一的职责，就是传达这种模式。剩下的实现细节，LLM 可以和你一起补完。
